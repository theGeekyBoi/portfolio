### 6.3 PCA Framework — Embedded Device Authentication via Principal Component Analysis
*Role: Developer / Researcher · Independent research (context to confirm — see §18)*

A from-scratch, C-native PCA pipeline for device authentication/fingerprinting from frequency-sweep sensor data, designed to run standalone on Arduino Q hardware with no tethered PC. The work spans numerical-methods implementation, embedded systems integration, and performance optimization.

**Core PCA pipeline & math**
- **C API + float32 kernels** — built the public C API alongside core float32 math kernels, intentionally keeping the architecture floating-point to preserve a straightforward path to eventual microcontroller (MCU) deployment.
- **Custom cyclic Jacobi eigensolver** — implemented natively in C to handle the eigendecomposition without external libraries.
- **Sign-ambiguity resolution** — deterministic sign normalization inside the eigensolver to mirror scikit-learn behavior, resolving PCA's inherent sign ambiguity.
- **Memory & scaling** — moved allocation to the heap to bypass a 256-feature limit, enabling the full **2001 data points per frequency sweep**.
- **Dual-covariance PCA** — switched to a dual-covariance formulation so the eigendecomposition scales with the number of samples rather than the feature count.
- **Simulation & validation** — wrote a Python exporter to generate simulated test vectors into a C header, validating the C pipeline's accuracy against simulated data.

**System architecture & workflow integration**
- **Full C port of `final_pca`** — recreated the original Python program's functionality entirely in C.
- **End-to-end automation** — natively loads CSV sweeps, constructs both single-sweep and multi-sweep PCA identifiers, executes device authentication, and outputs diagnostic CSV metrics alongside a final Markdown report.

**Hardware integration (Arduino Q)**
- **Circuit rebuild** — redid the physical measurement circuit and captured live hardware readings.
- **Standalone node architecture** — leveraged the Arduino Q's dual-environment setup to eliminate the need for a tethered PC:
  - *MCU side:* runs the actual frequency sweep and physical sensor measurements.
  - *Linux side:* collects data via the Bridge, formats it into datasets, runs the local C-based PCA pipeline, and generates final reports.
- **Version control** — packaged and pushed the complete hardware integration code to the `hardware` branch under the *Arduino Q Implementation* folder on GitHub.

**Performance optimization**
- Identified that primary latency came from the physical frequency sweeping, not the PCA computation, then:
  - **Reduced the data bottleneck** by batching multiple sweep points per Bridge call between MCU and Linux to minimize transfer overhead.
  - **Tuned the sweep** by lowering frequency values to cut the physical time per sweep.

| Metric | Initial setup | Optimized pipeline |
|---|---|---|
| Time per full sweep | 5–8 minutes | ~2 minutes |
| Total device flow (5 sweeps) | 27–40 minutes | 6.5–10 minutes |
